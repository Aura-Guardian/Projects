# Linux Packet Filtering Lab — iptables & ufw

A hands-on firewall configuration lab using raw Linux kernel packet filtering tools on Ubuntu Server 24.04, demonstrating deny-by-default security posture, stateful connection tracking, and dropped-packet logging.

---

## Objective

Demonstrate foundational host-based firewall configuration using both low-level (iptables) and high-level (ufw) interfaces to the Linux Netfilter packet filtering framework. Implement a deny-by-default policy, selectively allow necessary services, and generate structured logs for dropped traffic analysis.

---

## Key Concepts Covered

- Netfilter packet filtering framework (Linux kernel)
- INPUT / OUTPUT / FORWARD chain architecture
- Deny-by-default (allowlist) security policy
- Stateful connection tracking (ESTABLISHED, RELATED, NEW states)
- TCP vs UDP vs ICMP protocol-level rules
- Port-based access control (SSH, HTTP, HTTPS, DNS, NTP)
- Dropped packet logging with rate limiting
- Rule persistence across reboots
- Log analysis — reading UFW BLOCK entries and iptables kern.log entries
- DROP vs REJECT behavioral difference
- ufw as a frontend for iptables (understanding the relationship)

## What I Built

### Path A — Raw iptables

Configured Netfilter directly using iptables commands:

1. Flushed all existing rules for a clean state
2. Added SSH allow rule before enabling DROP (to avoid lockout)
3. Set default DROP policy on INPUT, OUTPUT, and FORWARD chains
4. Allowed loopback interface (required for OS internal communication)
5. Allowed ESTABLISHED and RELATED connections (stateful tracking)
6. Opened HTTP (80) and HTTPS (443) inbound with corresponding outbound reply rules
7. Allowed outbound DNS on UDP and TCP port 53
8. Added rate-limited LOG rules before the final DROP on each chain
9. Persisted rules across reboots using iptables-persistent

### Path B — ufw

Configured the same policy using ufw as a higher-level interface:

1. Reset ufw to factory defaults
2. Set default deny on incoming, outgoing, and forward
3. Allowed SSH inbound before enabling
4. Opened HTTP and HTTPS inbound
5. Allowed outbound DNS, HTTP, HTTPS, and NTP
6. Enabled medium-level logging
7. Activated the firewall and verified rules with `ufw status verbose`
8. Inspected the underlying iptables rules generated by ufw using `ufw show raw`

---

## Outcome / Findings

- Confirmed deny-by-default posture: nmap scans from Mac showed all non-allowed ports as `filtered` (packets silently dropped, no response)
- Confirmed allowed ports (22, 80, 443) showed as `open` in nmap scans
- Observed real-world scanning activity in logs within minutes of the lab — automated bots hit SSH (port 22) and common database ports (3306, 6379) repeatedly
- Verified that ufw generates equivalent iptables rules — `ufw show raw` output matched the manual iptables configuration from Path A
- Confirmed rule persistence: rules survived VM reboot via iptables-persistent (Path A) and ufw's built-in file storage (Path B)
- Demonstrated DROP vs REJECT difference: DROP caused 30+ second timeouts in curl/telnet, REJECT returned immediate "connection refused"
